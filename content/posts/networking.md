## Overview

The iGYM games use a **dedicated server architecture** built on **Unity Netcode for GameObjects (NGO)**. The system separates networking into two planes:

- **Control Plane** — A Python FastAPI lobby server handles matchmaking, room management, and dedicated server orchestration via REST API
- **Game Plane** — Unity Netcode for GameObjects handles real-time gameplay networking (physics, state sync, RPCs)

**Transport**: UnityTransport (UDP) for direct connections, with optional Unity Relay Service for online play

**Tick Rate**: 30 Hz (configurable via NetworkManager)

## Lobby System (network-api)

The lobby is a **Python FastAPI** REST server that manages rooms and spawns dedicated server processes. The Unity client and dedicated server each have their own C# SDK for communicating with the lobby.

### Client-Facing Endpoints

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/rooms?game_id=<id>` | List all rooms for a game (public info only) |
| POST | `/rooms` | Create a new room |
| POST | `/rooms/{room_id}/join` | Join an existing room |
| POST | `/rooms/{room_id}/leave` | Leave a room |
| POST | `/rooms/{room_id}/kick` | Kick a player (owner only) |
| GET | `/rooms/{room_id}` | Get full room state |
| POST | `/rooms/{room_id}/start` | Start room and spawn DS (owner only) |
| POST | `/rooms/{room_id}/connect` | Get DS connection info (IP/port/token) |

### Dedicated Server Endpoints

| Method | Endpoint | Purpose | Auth |
|--------|----------|---------|------|
| POST | `/rooms/{room_id}/ds/started` | DS reports it started listening | Bearer token |
| POST | `/rooms/{room_id}/ds/heartbeat` | DS sends periodic heartbeat | Bearer token |
| POST | `/rooms/{room_id}/ds/ended` | DS reports shutdown | Bearer token |

### Admin Endpoints

| Method | Endpoint | Purpose |
|--------|----------|---------|
| POST | `/rooms/{room_id}/end` | Force-end a room (owner only) |

### Room Management Flow

**Create Room:**
```
POST /rooms
{ "game_id": "igym_ar", "name": "Squad Battle", "capacity": 4, "private": false }
→ { "room_id": "a1b2c3d4" }
```

**Join Room:**
```
POST /rooms/a1b2c3d4/join
{ "client_guid": "device-uuid-12345678", "display_name": "Player One" }
→ RoomStateResp (includes player_id, token)
```

**Start Room (Owner Only):**
```
POST /rooms/a1b2c3d4/start
{ "owner_player_id": "12a3b4c5" }
→ ReadyConnectInfo with status="starting"
```

**Poll for Ready:**
```
POST /rooms/a1b2c3d4/connect
{ "client_guid": "device-uuid-12345678" }
→ ReadyConnectInfo (status="ready", server_ip, server_port, token)
```

### Authentication

**Client Identity:**
- `client_guid` — Stable 8–128 character identifier per app install
- Idempotent joins: same `client_guid` always maps to the same `player_id` in a room
- `player_id` — 8-char hex, generated per join, unique within room

**Player Token:**
- Format: `{SHARED_SECRET}:{room_id}:{player_id}`
- Returned in join/connect responses
- Client passes this token to the DS during connection approval
- DS validates the token without needing a database lookup

**DS Authentication:**
- Random 24-char URL-safe bearer token (`ds_secret`)
- Sent as `Authorization: Bearer <ds_secret>` header on all DS→lobby calls
- Generated by lobby at room start time

The lobby maintains a game registry with executable paths and launch configs:

## Dedicated Server Spawning & Lifecycle

When the room owner calls `POST /rooms/{id}/start`, the lobby:

1. Allocates a unique UDP port (starting from 7777, auto-increment)
2. Generates a random `ds_secret` bearer token
3. Spawns the Unity DS process with command-line arguments:

| Argument | Purpose |
|----------|---------|
| `-port` | UDP port to listen on |
| `-roomId` | Room identifier |
| `-maxPlayers` | Player capacity (2, 4, or 8) |
| `-ownerPlayerId` | Room owner's player ID |
| `-lobbyUrl` | Lobby callback URL for heartbeat/status |
| `-dsSecret` | Bearer token for lobby API auth |
| `-idleShutdownSeconds` | Seconds with no players before auto-quit (default: 10) |
| `-logFile` | Path for DS log output |

4. DS starts, binds to port, calls `POST /rooms/{id}/ds/started`
5. Lobby marks room as `ready`

**Heartbeat** (every 5 seconds):
```
POST /rooms/{room_id}/ds/heartbeat
{ "port": 7777, "player_count": 2, "match_running": true, "time_remaining": 450.5, "ts": ... }
```

**Heartbeat Timeout:** 20 seconds with no heartbeat → room marked `error`, lobby kills DS process

**Shutdown Reasons:** `empty_timeout`, `quit`, `error`

### Background Watcher

The lobby runs a daemon thread (every 2 seconds) that:
- Checks DS heartbeat timeouts
- Cleans up stale rooms (TTL: 1 hour)
- Detects DS process crashes
- Frees ports from ended rooms

## Connection Approval

When a client connects to the DS via Netcode, it sends a `ConnectPayload`:

```
ConnectPayload {
    sessionId   : string
    playerId    : string
    token       : string    // format: {SHARED_SECRET}:{room_id}:{player_id}
    displayName : string
    ts          : long      // timestamp
}
```

The server validates:
- Token matches the expected `{secret}:{roomId}:{playerId}` format
- Player belongs to the room
- Server capacity is not exceeded
- Caches room owner identity for GameManager

## Game State Synchronization

### NetworkVariables (Server → Clients)

Game-level state is pushed via NetworkVariables (write permission: server only):

| Variable | Type | Purpose |
|----------|------|---------|
| `GameManager.isRunning` | bool | Match active flag |
| `GameManager.syncedTime` | float | Countdown timer |
| `GameManager.syncedMode` | int | Difficulty: 0=Easy, 1=Medium, 2=Hard |
| `GameManager.syncedScoreLeft` | int | Left team score |
| `GameManager.syncedScoreRight` | int | Right team score |
| `GameManager.roomOwnerClientId` | ulong | Owner identity |
| `CircleController.displayName` | string | Player display name |

### Ball Position Sync

Ball position is synced via **HockeyNetworkTransform** (extends NetworkTransform):
- Server pushes authoritative position/rotation each tick
- Custom callback `OnNetworkTransformPositionUpdated()` feeds data into the snapshot buffer
- Clients interpolate between snapshots for smooth rendering

### Paddle Position Sync

Paddles are **client-owned NetworkObjects**:
- Position driven by mouse input (PositionSimulator) or ROS/CV (PosMessageNetworked)
- NetworkTransform interpolation disabled on owned paddles to prevent input lag
- Spawned/despawned via PaddleSpawnBridge RPCs (server-authoritative)

## Client-Side Prediction & Reconciliation

### Ball Visual States

The ball controller (`hockeyctr.cs`) operates in three visual states:

{{< mermaid >}}

    A -->|"Paddle hits ball"| P

    P -->|"Server ACK, error ≥ 0.05"| R
    R -->|"Error drops below 0.05"| A

    P2["Sends ApplyVelocityServerRpc\nSets prediction gate ~1 RTT"]
    R2["Exponential blend, half-life 80ms\nMax correction speed 20 units/sec"]

    P -.- P2
    R -.- R2

### Prediction Flow

```
[Paddle Collision Detected]
        ↓
[Calculate hit velocity from paddle movement]
        ↓
[PredictHit(velocity)]
  - Set state = Predicting
  - Store predicted velocity and position
  - Set prediction gate (block server updates for ~1 RTT)
        ↓
[Send ApplyVelocityServerRpc(velocity, hitId)]
        ↓
[Server applies: _rb.linearVelocity = velocity]
  - Broadcasts position via NetworkTransform
  - Sends AcknowledgeHitClientRpc(hitId)
        ↓
[Client receives server updates]
  - If predicted ≈ true position → snap to Authority
  - If diverged → enter Reconciling state
```

### Reconciliation Parameters

| Parameter | Value | Purpose |
|-----------|-------|---------|
| `reconcileSnapThreshold` | 0.05 units | Snap to authority below this distance |
| `reconcileHalfLifeSec` | 0.08 s | Exponential blend half-life |
| `reconcileMaxSpeed` | 20 units/s | Max correction speed |

### Interpolation (Snapshot Buffer)

For rendering the ball between server updates:
- Circular buffer stores up to 128 snapshots `{ receiveTime, tick, position }`
- **Adaptive render delay** based on network conditions:
  - Formula: `oneWayRtt * 0.35 + jitterEWMA * 0.5 + skippedTicksPenalty`
  - Bounds: min 20ms, max 250ms
- Linear interpolation between the two surrounding snapshots

## Lobby Server Configuration

Key configuration values in the lobby server (`lobby.py`):

| Setting | Default | Purpose |
|---------|---------|---------|
| `DS_BASE_PORT` | 7777 | Starting port for DS allocation |
| `PUBLIC_LOBBY_URL` | `http://127.0.0.1:8080` | DS heartbeat callback URL |
| `SHARED_SECRET` | `"dev_secret_change_me"` | Token signing secret (**change in production**) |
| `ROOM_TTL_SEC` | 3600 | Stale room auto-delete (seconds) |
| `DS_HEARTBEAT_TIMEOUT_SEC` | 20 | Seconds before room marked error |
| `DS_HEARTBEAT_INTERVAL_SEC` | 5 | DS heartbeat frequency (seconds) |
| `DS_EMPTY_SHUTDOWN_GRACE_SEC` | 10 | Idle seconds before DS auto-quits |
| `SERVER_VIEW` | True | Show spectator window on DS |

## Runtime Debug Toggles

The **NetworkFeaturesController** provides runtime toggles accessible during gameplay:
- **Prediction On/Off** — Enable or disable client-side prediction
- **Interpolation On/Off** — Enable or disable snapshot interpolation

These are useful for debugging network behavior and measuring the impact of prediction/interpolation on gameplay feel.
